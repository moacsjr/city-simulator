Engenharia de Simulação Histórica e Computação Gráfica em Tempo Real: O Projeto "Medieval City Evolution"A criação de um ambiente virtual capaz de simular a evolução contínua e orgânica de um assentamento humano ao longo de séculos coloca desafios multidisciplinares complexos. O projeto "Medieval City Evolution" estabelece como meta o desenvolvimento de uma narrativa visual ininterrupta controlada por um único controlo deslizante de progresso ($0\%$ a $100\%$). Ao contrário dos sistemas tradicionais baseados na alternância de cenas discretas, este motor gráfico propõe uma transição sem quebras estruturais, onde cada elemento arquitetónico, habitante, condição atmosférica e paisagem sonora reage de forma analógica e fluida à manipulação da linha temporal.Teoria do Crescimento Orgânico e Morfologia Urbana MedievalPara que a evolução do assentamento resulte numa experiência visualmente credível, a cidade virtual deve obedecer aos princípios históricos do urbanismo orgânico europeu desenvolvido entre os séculos IX e XII. A morfogénese destas cidades demonstra que os aglomerados medievais não surgiam de forma geométrica planeada, exceto em fundações específicas de caráter militar ou comercial, como as bastides francesas. Pelo contrário, as cidades orgânicas evoluíam de forma incremental a partir de um núcleo ou polo atrator de grande relevância local.A simulação adota um modelo de crescimento centrado no desenvolvimento de núcleos sucessivos de atração e infraestrutura. O assentamento inicia-se em torno de um recurso natural vital — um rio navegável ou um entroncamento de estradas comerciais. Com o aumento da densidade populacional, ergue-se uma praça central para trocas comerciais diretas.A partir deste centro cívico, estruturam-se bairros especializados através de uma malha radial-concêntrica: zonas habitacionais densas, o distrito religioso liderado pela catedral, áreas industriais (como oficinas e moinhos) e o setor agrícola periférico. Cada elemento na simulação é instanciado como um objeto persistente associado a regras de interpolação matemática estritas, cuja evolução dimensional e de opacidade depende diretamente da variável global de progresso $p \in [0, 100]$.Funções de Ativação e Suavização de EscalaCada objeto tridimensional ativo é mapeado com atributos individuais que determinam o seu ciclo de vida. Para garantir que o nascimento e maturação de uma estrutura não ocorram de forma abrupta, utiliza-se uma curva cúbica de suavização baseada na formulação matemática de smoothstep. Definindo os limites de início e fim de desenvolvimento de uma estrutura como $growStart_i$ e $growEnd_i$, o fator de normalização local $x_i(p)$ e a respetiva escala de renderização $s_i(p)$ são calculados da seguinte forma:$$x_i(p) = \text{clamp}\left( \frac{p - growStart_i}{growEnd_i - growStart_i}, 0, 1 \right)$$$$\alpha_i(p) = 3x_i(p)^2 - 2x_i(p)^3$$$$s_i(p) = \alpha_i(p) \cdot S_{max, i}$$Esta equação garante uma taxa de crescimento contínua com aceleração gradual e desaceleração suave nas extremidades temporais de desenvolvimento de cada modelo tridimensional, conferindo uma transição natural.A opacidade $\theta_i(p)$ e a transição cromática de texturas (como a mudança de cor do telhado de madeira para cerâmica) utilizam interpolações lineares (lerp) baseadas no mesmo fator normalizado:$$\theta_i(p) = \text{clamp}\left( \frac{p - appearAt_i}{growStart_i - appearAt_i}, 0, 1 \right)$$$$\text{CorRoof}(p) = (1 - x_i(p)) \cdot \vec{C}_{\text{wood}} + x_i(p) \cdot \vec{C}_{\text{tile}}$$A listagem de código seguinte demonstra a estrutura de dados e a classe lógica em JavaScript para gerir a atualização de cada objeto da cidade tridimensional em resposta ao controlo global de progresso:JavaScriptclass EvolutiveObject {
    constructor(config) {
        this.id = config.id;
        this.appearAt = config.appearAt;
        this.growStart = config.growStart;
        this.growEnd = config.growEnd;
        this.maxScale = config.maxScale;
        this.mesh = config.mesh; // Referência para o objeto tridimensional na cena Three.js
        this.colorTransitions = config.colorTransitions || null;
    }

    update(progress) {
        // Gestão de visibilidade primária baseada nos limites de progresso
        if (progress < this.appearAt) {
            this.mesh.visible = false;
            return;
        }
        
        this.mesh.visible = true;

        // Cálculo da interpolação de escala usando Smoothstep
        let localFactor = 0;
        if (progress >= this.growStart) {
            const range = this.growEnd - this.growStart;
            localFactor = Math.min(1, Math.max(0, (progress - this.growStart) / range));
        }

        // Aplicação do polinómio cúbical smoothstep
        const smoothFactor = 3 * Math.pow(localFactor, 2) - 2 * Math.pow(localFactor, 3);
        const scaleVal = smoothFactor * this.maxScale;
        this.mesh.scale.set(scaleVal, scaleVal, scaleVal); // [cite: 11]

        // Cálculo de opacidade na fase inicial de aparecimento
        let opacityVal = 1;
        if (progress < this.growStart && this.growStart !== this.appearAt) {
            opacityVal = (progress - this.appearAt) / (this.growStart - this.appearAt);
        }
        
        // Aplicação de opacidade ao material do objeto
        if (this.mesh.material) {
            const materials = Array.isArray(this.mesh.material) ? this.mesh.material : [this.mesh.material];
            materials.forEach(mat => {
                mat.transparent = true;
                mat.opacity = opacityVal;
            });
        }

        // Interpolação dinâmica de cor/textura se aplicável
        if (this.colorTransitions && localFactor > 0) {
            const startColor = new THREE.Color(this.colorTransitions.from);
            const endColor = new THREE.Color(this.colorTransitions.to);
            const lerpedColor = startColor.lerp(endColor, localFactor);
            
            if (this.mesh.material && this.mesh.material.color) {
                this.mesh.material.color.copy(lerpedColor);
            }
        }

        // Atualização da matriz de transformação local do objeto na cena
        this.mesh.updateMatrix(); // [cite: 11]
    }
}
Engenharia Gráfica e Otimização com Three.jsA exibição de uma simulação tridimensional interativa contendo trezentos habitantes com inteligência de movimentação autónoma e centenas de elementos arquitetónicos em navegadores web esbarra frequentemente no custo computacional associado às chamadas de desenho (draw calls) dirigidas à GPU.A análise comparativa entre as diferentes arquiteturas gráficas (Three.js, PixiJS, Fabric.js) demonstra que, embora o motor PixiJS seja mais veloz para grandes volumes de elementos planos devido à sua tecnologia de empacotamento automático de texturas (automatic draw call batching), o seu ecossistema puramente bidimensional inviabiliza a representação volumétrica com iluminação realista de tipo PBR, essencial para a verosimilhança do assentamento medieval.Por sua vez, o Fabric.js revela um desempenho inadequado para a gestão de simulações em larga escala devido ao processamento assente no contexto de canvas 2D. Deste modo, o Three.js, especialmente se associado ao novo renderizador WebGPU, destaca-se como a única ferramenta viável para a execução das transformações volumétricas tridimensionais exigidas pelo projeto.Renderização Eficiente via InstancedMeshPara garantir que o projeto funcione fluidamente a $60\text{ FPS}$ em computadores domésticos e dispositivos móveis, é impensável renderizar cada edifício ou árvore como um objeto geométrico isolado na cena, visto que isso geraria milhares de chamadas de desenho em tempo real. A solução reside no uso intensivo da técnica de renderização instanciada por intermédio da classe THREE.InstancedMesh.Esta API permite agrupar múltiplos ativos idênticos partilhando a mesma geometria e material (como tijolos de muralha, estacas de vedação, árvores ou cavalos) sob uma única chamada de renderização unificada. O Algoritmo de Desvanecimento Cruzado de Potência EquivalenteUm desvanecimento cruzado (crossfade) linear simples entre duas pistas de áudio gera uma quebra acentuada do ganho combinado de pressão sonora na zona intermédia de transição ($50\%$), o que é interpretado pelo ouvido humano como uma quebra no volume geral. Para anular este efeito físico e manter a intensidade sonora combinada constante ao longo de todo o espectro da simulação, aplica-se a função de desvanecimento de potência equivalente (equal-power crossfade).O modelo calcula as funções de ganho $\gamma_1(p)$ e $\gamma_2(p)$ para duas fontes sonoras vizinhas utilizando funções trigonométricas projetadas sobre a normalização da percentagem de progresso para o intervalo $[0, 1]$:$$\gamma_1(p) = \cos\left( p \cdot \frac{\pi}{2} \right)$$$$\gamma_2(p) = \sin\left( p \cdot \frac{\pi}{2} \right) = \cos\left( (1 - p) \cdot \frac{\pi}{2} \right)$$Como a identidade pitagórica assegura que a soma dos quadrados das funções trigonométricas básicas de um ângulo é constante ($\cos^2\theta + \sin^2\theta = 1$), a soma energética das duas saídas de sinal mantém-se perfeitamente equilibrada e linear aos ouvidos do utilizador, impossibilitando quedas ou picos artificiais de volume.O código a seguir apresenta a infraestrutura de áudio desenvolvida em conformidade com as regras de agendamento de sinal de precisão temporal da Web Audio API:JavaScriptconst audioContext = new (window.AudioContext || window.webkitAudioContext)();

class SeamlessSoundtrack {
    constructor(sourcesList) {
        this.sources = sourcesList; // Array de objetos contendo caminho do ficheiro e limites
        this.nodes = {};
        this.masterGain = audioContext.createGain();
        this.masterGain.gain.setValueAtTime(0.8, audioContext.currentTime); // Proteção contra saturação
        this.masterGain.connect(audioContext.destination); // [cite: 23]
    }

    async initialize() {
        for (const track of this.sources) {
            const response = await fetch(track.url);
            const arrayBuffer = await response.arrayBuffer();
            const audioBuffer = await audioContext.decodeAudioData(arrayBuffer); //

            const bufferSource = audioContext.createBufferSource(); //
            bufferSource.buffer = audioBuffer;
            bufferSource.loop = true;

            const gainNode = audioContext.createGain();
            gainNode.gain.setValueAtTime(0, audioContext.currentTime); //

            // Ligação do fluxo: Source -> Gain -> Master -> Destino [cite: 23]
            bufferSource.connect(gainNode);
            gainNode.connect(this.masterGain);

            this.nodes[track.id] = {
                source: bufferSource,
                gain: gainNode,
                activeRange: track.range
            };

            // Inicia a reprodução silenciosa em loop
            bufferSource.start(0); //
        }
    }

    updateVolume(progress) {
        if (audioContext.state === 'suspended') {
            audioContext.resume();
        }

        const t = progress / 100; // Normalização para intervalo [0, 1]

        for (const [id, node] of Object.entries(this.nodes)) {
            const [minRange, maxRange] = node.activeRange;
            
            if (t >= minRange && t <= maxRange) {
                // Cálculo do fator local normalizado para a zona ativa do som
                const span = maxRange - minRange;
                const localProgress = (t - minRange) / span;

                // Aplicação da curva matemática de potência equivalente
                const gainVolume = Math.sin(localProgress * Math.PI * 0.5); // [cite: 23, 25, 27]
                node.gain.gain.setValueAtTime(gainVolume, audioContext.currentTime); //
            } else if (t < minRange && t >= minRange - 0.2) {
                // Curva de desvanecimento de saída para a pista anterior (cosseno) [cite: 25, 26]
                const dist = minRange - t;
                const normalizedDist = dist / 0.2;
                const gainVolume = Math.cos(normalizedDist * Math.PI * 0.5); //
                node.gain.gain.setValueAtTime(gainVolume, audioContext.currentTime);
            } else {
                // Silêncio absoluto para pistas fora da sua zona de influência temporal
                node.gain.gain.setValueAtTime(0, audioContext.currentTime); //
            }
        }
    }
}

Com base nas restrições de desempenho e morfologia histórica analisadas, formulam-se as seguintes diretrizes para implementação:Adoção Estrita de Renderização Instanciada: A totalidade dos objetos decorativos repetitivos (árvores, habitações comuns, muros, animais) deve ser processada através de estruturas THREE.InstancedMesh para otimizar as chamadas de desenho (draw calls). Apenas edifícios singulares e de grandes dimensões (Catedral, Castelo, Prefeitura) devem ser geridos como elementos geométricos individuais na cena tridimensional.Segregação Espacial e Frustum Culling: Devido à densidade de trezentos habitantes no estágio de Metrópole, a cena deve ser estruturada em células espaciais. O descarte de instâncias invisíveis à câmara (frustum culling) deve ser gerido de forma otimizada para evitar que a GPU processe vértices desnecessários posicionados fora do campo visual do utilizador.Transição de Texturas em Sombreadores: Evitar a alteração dinâmica de materiais base das instâncias para impedir quebras de desempenho devido à recompilação de código WebGL (shaders). Todas as variações físicas dos materiais (como a transição de telhados de madeira para cerâmica, ou caminhos de terra para calçada) devem ser codificadas em mapas de texturas partilhados, interpolados linearmente nos sombreadores a partir de uma variável uniforme de progresso temporal.Preservação da Energia Acústica: A mistura de efeitos sonoros ambientes deve recorrer obrigatoriamente a nós de ganho coordenados através do modelo de potência equivalente baseada em funções trigonométricas da Web Audio API. Isto assegura a estabilidade da pressão sonora durante todo o processo de evolução da cidade, impedindo desníveis de som incómodos para o utilizador.
